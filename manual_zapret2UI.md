# Zapret2UI — полный мануал

Подробный справочник по **Zapret2UI** — графической оболочке для движка обхода блокировок
[zapret2](https://github.com/bol-van/zapret2) (`winws2`). Здесь разобраны все функции, вкладки,
настройки, встроенные стратегии, формат своих стратегий, встроенный Telegram-прокси, файлы на диске
и решение проблем.

Если нужно быстро начать — читайте [README](README.md). Этот документ — для тех, кто хочет понять
программу целиком и настраивать её тонко.

> Мануал по **приложению**. Документация по самому движку (полный список desync-глаголов и флагов
> `winws2`) — официальная:
> [bol-van/zapret2 · manual.en.md](https://github.com/bol-van/zapret2/blob/master/docs/manual.en.md).

## 1. Что это и на что способно

Zapret2UI не «ломает» блокировки на сервере и ничего не меняет на самих сайтах. Он управляет
движком `winws2`, который на лету слегка изменяет первые сетевые пакеты ваших соединений так, что
оборудование провайдера (**DPI/ТСПУ**) не распознаёт, к какому сайту вы подключаетесь, а сам сервер
понимает всё правильно. Набор таких приёмов называется **стратегией**.

Возможности:

- Обход блокировок/замедлений для **Discord** и **YouTube** одной кнопкой.
- Встроенный **Telegram-прокси** (MTProto → WebSocket через Cloudflare) — обходит блокировку Telegram
  по IP; работает отдельно и **без прав администратора**.
- **9 готовых стратегий**, **автоподбор** лучшей и **генерация** личной стратегии под провайдера.
- **Память по сетям**: рабочая стратегия запоминается для каждой сети и включается сама.
- **Авто-починка**: если обход отвалился, программа переподбирает рабочий вариант.
- **Диагностика**: таблица доступности по сервисам.
- **Свои хостлисты и цели**, **обход по IP** (ipset), **игровой фильтр**, **отключение QUIC**.
- **Один self-contained `.exe`**, движок докачивается при первом запуске и проверяется по SHA-256.

## 2. Требования и установка

**Требования:**

- Windows 10 или 11, **x64** (есть и x86-вариант движка, но основной — 64-битный).
- **Права администратора** для обхода (движок грузит драйвер WinDivert в ядро). Для Telegram-прокси
  админ не нужен.
- Доступ в интернет при первом запуске (докачать движок).

**Установка:**

1. Скачайте `Zapret2UI.exe` со [страницы релизов](https://github.com/Asterlike/zapret2UI/releases/latest).
   Это один файл — установка не требуется, .NET ставить не нужно.
2. Запустите **от имени администратора** (правой кнопкой → «Запуск от имени администратора», либо
   согласитесь на запрос UAC).
3. При первом запуске программа докачает движок `winws2` в `%LOCALAPPDATA%\Zapret2UI\engine` и проверит
   загрузку по контрольной сумме **SHA-256**.

Движок и приложение обновляются раздельно (см. [§15](#15-обновления)).

## 3. Как работает обход (детально)

- **SNI.** В начале почти любого TLS-соединения браузер открытым текстом отправляет имя сайта (поле
  **SNI** в ClientHello). DPI читает его и по чёрному списку обрывает соединение (RST, «зависание»,
  подмену).
- **Desync.** `winws2` перехватывает исходящие пакеты через **WinDivert** и применяет к ним desync:
  разрезает ClientHello, вставляет «фейковые» пакеты, играет с TTL/checksum/seq/ack/timestamps —
  чтобы DPI «увидел» не то, что реально уходит на сервер. Правильно настроенный фейк **умирает** до
  сервера (не долетает), а реальный пакет доходит.
- **Почему нет одной стратегии на всех.** Разные провайдеры фильтруют по-разному, поэтому приём,
  пробивающий блокировку у одного, бесполезен у другого. Отсюда **подбор** и **генерация**.
- **Allow-list по умолчанию.** Программа обходит только то, что в списках (Discord/YouTube + ваши
  цели), а не весь трафик — чтобы не задеть игры и приложения (см. [§7](#7-область-обхода-игровой-фильтр-quic)).
- **Чего обход не умеет.** Он снимает блокировку **по имени** сайта. Если ресурс режут **по IP** —
  поможет только VPN или (для Telegram) встроенный прокси.

## 4. Интерфейс: режимы и вкладки

Вверху по центру — переключатель **Простой / Расширенный**.

- **Простой** — одна большая кнопка «Включить обход», карточка Telegram и выбор цели. Больше ничего.
- **Расширенный** — семь вкладок: **Главная, Стратегии, Хостлисты, Диагностика, Журнал, Telegram,
  Настройки**.

### 4.1. Главная

- **Кнопка «Включить / Выключить обход»** — старт/стоп движка. Над ней точка и подпись состояния:
  `Остановлен` → `Запуск…` → `Работает` (и `Остановка…`).
- **Цель обхода** («ЧТО ПОДОБРАТЬ»): `Discord + YouTube`, `Discord` или `YouTube` — что программа
  старается открыть при подборе/генерации.
- **Карточка Telegram** — адрес и секрет встроенного прокси, кнопки «Открыть» и «Копировать».
- **Тумблер «Telegram через прокси»** — включает/выключает прокси (дублирует вкладку Telegram).
- **«Подобрать стратегию»** — перебирает проверенные стратегии и оставляет ту, что открывает выбранные
  цели. Начинайте с неё: идёт по готовому списку и потому предсказуемее.
- **«Сгенерировать стратегию»** — собирает вариант с нуля, отдельно под Discord и YouTube. Дольше,
  нужна тогда, когда подбор не пробил.
- QR и ссылка **«Поддержать автора»**.

### 4.2. Стратегии

Список готовых способов обхода. Выберите строку → **«Применить»** (обход перезапустится на ней).
Наведите курсор — во всплывающей подсказке полное имя и описание. Рекомендуемая стратегия помечена и
включается по умолчанию. Здесь же появляются **ваши** стратегии, сохранённые после подбора/генерации.
Подробно о наборе — [§5](#5-встроенные-стратегии).

### 4.3. Хостлисты

Списки доменов (по одному на строку — формат `winws2`), к которым применяется обход: `youtube`,
`discord` и другие. Встроенные списки обновляются сами при каждом запуске; ваши строки не трогаются.
Можно создать свой список, отредактировать и выбрать активный. Подробно — [§8](#8-хостлисты-свои-цели-и-обход-по-ip).

### 4.4. Диагностика

Слева — кнопки, справа — **таблица доступности**: строки по сервисам (Discord: вход/API, Gateway,
CDN, обновления, CF-челлендж; YouTube: web, короткие ссылки, картинки, видео-редирект; плюс Google,
Cloudflare, DNS), столбцы — HTTP / TLS 1.2 / TLS 1.3 / Ping. Зелёная ячейка — открывается, красная —
режется. Кнопки:

- **Подобрать лучшую** — перебирает готовые стратегии, проверяет доступность и оставляет лучшую.
- **Сгенерировать стратегию** — собирает **личную** стратегию под вашу сеть (дольше, точнее).
- **Диагностика** — просто проверить, что открывается (ничего не меняет).
- **Проверка DPI** — определить, **режет ли провайдер именно по DPI** (по имени сайта или по объёму/пакетам), а не «сайт недоступен» (см. ниже).
- **Свои цели** — добавить домен, чтобы его тоже проверяли и обходили.

Подбор/генерация запускают движок и требуют прав администратора. Подробно — [§6](#6-автоподбор-и-генерация).

**Как работает «Проверка DPI».** Обычная диагностика отвечает «открывается / не открывается». Проверка
DPI отвечает на другой вопрос — **вмешивается ли провайдер на уровне DPI**. Для каждого чувствительного
хоста (`discord.com`, `gateway.discord.gg`, `cdn.discordapp.com`, `www.youtube.com`) выполняется два шага:

1. **TCP-подключение** к порту 443. Если оно не проходит — это не DPI по имени сайта, а недоступность
   (маршрут или блок по IP); так и пишется — «нет соединения».
2. Если TCP-подключение прошло (сервер **принял** соединение — значит он живой), отправляется **TLS
   ClientHello с реальным именем сайта (SNI)**, и смотрится, что происходит именно с этим пакетом:
   - **обрыв (RST)** сразу после ClientHello — провайдерский DPI **вбросил сброс соединения** (классическая
     российская сигнатура): сервер уже согласился на TCP, поэтому сброс именно на пакете с именем сайта —
     это не сервер, а «железка» в середине;
   - **заморозка** — ClientHello ушёл, ответа нет, соединение висит: пакет с SNI **тихо дропается**;
   - **рукопожатие прошло** — DPI не вмешивается («чисто»).

Итог показывается вердиктом: «провайдер режет по DPI: обрыв/заморозка» (лечится обходом), «признаков DPI
нет» или «нет соединения». Точнее всего запускать **с выключенным обходом** — тогда видно «сырое»
поведение провайдера; при включённом обходе «чисто» означает, что обход уже снял DPI.

**Второй вид проверки — лимит по объёму/пакетам (метод «TCP 16-20»).** Кроме блокировки по имени сайта
провайдер может **душить само соединение по количеству пакетов**: TCP-подключение и первые десятки
килобайт проходят нормально, а дальше поток **замирает** (≈16–20 пакетов в каждую сторону). Это и есть
«сайт открывается, но видео буферит / медиа не грузятся / связь отваливается на больших объёмах» —
короткое TLS-рукопожатие из проверки выше под лимит не попадает и показывается как «чисто». Поэтому
приложение отдельно **тянет крупную загрузку** (через служебный тестовый канал) и смотрит: если поток
спокойно проходит за порог — лимита нет; если **замирает на первых десятках КБ** — это метод «TCP 16-20».
Важно: такой лимит по объёму TLS/SNI-обход снимает **не всегда** — иногда помогает только смена стратегии
или сети. Идея метода — из проекта [hyperion-cs/dpi-checkers](https://github.com/hyperion-cs/dpi-checkers)
(Apache-2.0).

### 4.5. Журнал

режим. Логи также пишутся в файлы (см. [§16](#16-файлы-и-папки-на-диске)).

## 5. Встроенные стратегии

| Стратегия | Для чего / что внутри |
|---|---|
| **Комбо (рекомендуемый)** | Включается по умолчанию. Лучшее под каждый сервис в одной команде, маршрутизация по SNI: Discord → `hostfakesplit` (быстрый вход + медиа), YouTube/Google → `fake`+`multidisorder`, остальное → `hostfakesplit`. Голос: STUN + RTP-фикс. Начинать с него. |
| **Комбо — отечественный (VK, целевой)** | Под РФ: фейки для Discord маскируются под `vk.com` (ТСПУ не роняет отечественный трафик). Как и рекомендуемый — `hostfakesplit` по маркеру SNI, поэтому пускает и в логин, и к серверам. Берите, если google-варианты «зелёные, но Discord не открывают». |
| **Комбо — Flowseal ALT10 (двойной fake + ts)** | Перевод стратегии Flowseal general (ALT10) на nfqws2: без разрезки — двойной fake-пакет (google + vk-ClientHello) с ts-fooling, голос через отечественный QUIC-блоб. У многих «работает вообще всё». |
| **Комбо — Flowseal ALT11 (fake+ts → seqovl)** | Второй вариант Flowseal: fake-прайм с ts + `multisplit` с большим `seqovl` (реальный google-ClientHello как паттерн). Если ALT10 не зашёл — пробуйте этот. |
| **Комбо — Flowseal (multisplit seqovl)** | Рабочий профиль на разрезке пакета с `seqovl`. |
| **Комбо — Flowseal ALT (fake+fakedsplit)** | Ещё один Flowseal: `fake` с `tcp_ts` + `fakedsplit`. Пробуйте, если вариант с `multisplit` не пробил. |
| **Комбо — окно (wssize)** | Обход через уменьшение TCP-окна (`wssize`) — помогает на части провайдеров. |
| **Discord — голос (QUIC-фейк)** | Если текст в Discord есть, а голос «подключается, но не слышно». |
| **Discord — адаптивный (circular, эксперим.)** | Только под Discord. Оркестратор `circular` сам чередует стратегии на лету (hostfakesplit → двойной fake → seqovl), ловя RST/ретрансмиссию, и оседает на первой рабочей. Дайте ему несколько секунд после старта. Требует загруженной библиотеки `zapret-auto.lua` (загружается автоматически). |

Порядок перебора вручную при проблемах — сверху вниз, как в таблице. Голос отдельно — `Discord —
голос (QUIC-фейк)`.

## 6. Автоподбор и генерация

Обе функции — на вкладке **Диагностика**; во время работы открывается окно с живым прогрессом.

- **Подобрать лучшую** (быстро). Программа по очереди запускает кандидатов из встроенного каталога,
  для каждого проверяет доступность целевых хостов (TLS 1.2 и 1.3), считает число неудач и выбирает
  вариант с наименьшим их числом. Итог можно **применить** и **сохранить как стратегию**.
- **Сгенерировать стратегию** (точнее, дольше). Программа тестирует отдельные компоненты обхода,
  оптимизирует Discord и YouTube по отдельности, затем собирает их в одно комбо и проверяет сборку
  целиком. На выходе — **личная стратегия** под вашу сеть, её тоже можно сохранить.

Цель (Discord+YouTube / Discord / YouTube) задаётся переключателем и определяет, какие хосты
считаются «целевыми». Найденная стратегия привязывается к текущей сети (см. [§13](#13-память-по-сетям)).

> Важно: и подбор, и диагностика проверяют доступность на низком уровне. «Зелёная» проверка не всегда
> означает, что сайт откроется в браузере — сайт может резаться иначе (ECH, блокировка по IP). Если
> проверка зелёная, а сайт не работает — пробуйте другую стратегию, отключите QUIC или, при блокировке
> по IP, VPN.

## 7. Область обхода, игровой фильтр, QUIC

Три настройки на вкладке **Настройки** определяют, *что именно* трогает движок.

- **Область обхода** (`BypassAllSites`):
  - **Только списки** (по умолчанию, безопасно, как у Flowseal) — обходятся лишь домены из хостлистов
    (YouTube/Discord/Telegram) + ваши цели. «Catch-all»-профили (обход всего остального TLS/QUIC)
    пере-нацеливаются на ваши цели или отбрасываются, поэтому игры и приложения не из списков **не
    задеваются**.
  - **Все сайты** — обходится весь TLS/QUIC (кроме списка исключений: банки, госуслуги и т.п.).
    Удобно, но может сломать игру/приложение, которого нет в исключениях.
- **Игровой фильтр** (`GameFilter`): расширяет перехват на все высокие порты (>1023), чтобы обход
  доставал и до замедляемых игр. По умолчанию выключен — захват узкий (80,443 + голосовой диапазон
  Discord), игры идут напрямую.
- **QUIC / HTTP-3** (`DisableQuic`): при включении QUIC у обходимых сервисов **дропается**, браузер
  откатывается на TCP/H2. Включайте, если провайдер режет/замедляет QUIC (YouTube «буферизует» по
  HTTP-3, но нормально работает по TCP).

Технически ширина перехвата задаётся токенами `{WF_TCP}`/`{WF_UDP}` (см. [§9](#9-токены-стратегий));
захват TCP всегда `80,443-65535`, чтобы не терять медиа/CDN Discord на высоких портах.

## 8. Хостлисты, свои цели и обход по IP

- **Хостлист** — текстовый файл, по одному домену на строку (`youtube.com`, `discord.gg`, …). Именно
  так их читает `winws2`. Встроенные списки (`youtube`, `discord`) обновляются при каждом запуске;
  свои — не трогаются.
- **Активный хостлист** подставляется в стратегию токеном `{HOSTLIST}`. Именованные списки —
  `{HOSTLIST:name}` (файл `lists\name.txt`).
- **Свои цели** (кнопка на «Диагностике») — добавляют домен в отдельный агрегированный список; он
  учитывается при подборе/генерации и обходится даже в режиме «только списки». При добавлении
  учитываются поддомены и домены того же бренда на других зонах (например, `yandex.ru` →
  `yandex.kz`/`ya.ru`).
- **Обход по IP (ipset).** Если ресурс режут по IP-адресам, обход по доменам не помогает. На вкладке
  Telegram/Диагностике есть **«Собрать IP-список Discord»**: программа резолвит домены Discord через
  `mdig.exe | ip2net.exe` и собирает актуальные подсети (CIDR) в `lists\ipset-discord.txt` (прав
  администратора не нужно — только DNS). Подключается в стратегии токеном `{IPSET}` или
  `{IPSET:name}`.

## 9. Токены стратегий

Стратегии хранятся с токенами, которые движок-сервис (`EngineService.BuildArguments`) разворачивает
при запуске в реальные аргументы `winws2`:

| Токен | Разворачивается в |
|---|---|
| `{FILES}` | путь `engine\files` (блобы: `--blob=x:@{FILES}\fake\stun.bin`) |
| `{WF}` | путь `engine\windivert.filter` (`--wf-raw-part=@{WF}\windivert_part.*.txt`) |
| `{WF_TCP}` | `--wf-tcp-out=80,443-65535` (ширина TCP-перехвата) |
| `{WF_UDP}` | `--wf-udp-out=…` (443 + STUN + голос Discord; игровой фильтр расширяет до `443-65535`) |
| `{HOSTLIST}` | `--hostlist=<активный список>` или пусто |
| `{HOSTLIST:name}` | `--hostlist=lists\name.txt` (или пусто, если файла нет) |
| `{EXCLUDE:name}` | `--hostlist-exclude=lists\name.txt` (защита банков/госуслуг от catch-all) |
| `{IPSET}` | `--ipset=<ipset-discord.txt>`, если IP-список собран, иначе пусто |
| `{IPSET:name}` | `--ipset=lists\ipset-name.txt` (или пусто) |

Пустой токен (например, `{HOSTLIST}` без активного списка) просто выкидывается из команды.

## 10. Написание своих стратегий

Стратегию можно создать/отредактировать вручную (сохраняется в `presets.json`). Это язык аргументов
`winws2`. Ниже — практические правила, проверенные на локальном движке; полный справочник глаголов —
в [официальном мануале](https://github.com/bol-van/zapret2/blob/master/docs/manual.en.md).

**Структура команды:**

- Профиль начинается с `--new`. Первый сегмент (до первого `--new`) несёт глобальную настройку
  (`--wf-*`, `--blob=`, `--ipcache-*`) плюс первый профиль.
- Фильтры профиля: `--filter-tcp=`/`--filter-udp=` (порты), `--filter-l7=tls,quic,http,discord,stun`
  (протокол), `{HOSTLIST}`/`{IPSET}` (по кому применять). Порядок этих флагов не важен.
- **Порядок важен для «липких» флагов:** `--payload=`, `--in-range=`, `--out-range=` действуют с
  места указания и до переопределения — их ставят **перед** нужным `--lua-desync=`.
- `--wf-tcp-out`/`--wf-udp-out` — порты через запятую в **одном** флаге (повтор перезаписывает).

**Глаголы desync** (`--lua-desync=`), подтверждённые в движке:

- `fake` — вставить фейковый пакет (`blob=<имя>`). Fooling применяется **только к фейку** — реальный
  пакет доходит чистым. Хорош для гейтвея.
- `multisplit` / `multidisorder` / `multidisorder_legacy` — разрезать (и переупорядочить) реальный
  пакет. Fooling применяется **ко ВСЕМ сегментам** — сюда **нельзя** класть `tcp_ack`/`badseq`/битый
  ttl (сломает реальное соединение), только `pos`/`seqovl` (+ опц. `ip_id`, `tcp_ts_up`).
- `hostfakesplit` — разрез по маркеру SNI + фейковый host той же длины; адаптируется к размеру
  ClientHello, поэтому дружелюбен к гейтвею Discord. Fooling обязателен.
- `fakedsplit` / `fakeddisorder` — разрез с фейком; fooling только на фейках.
- `syndata` — данные в SYN. Без разрушающего fooling (`tcp_seq`/`tcp_ack`/`badsum` ломают рукопожатие).
- `tcpseg` — один сегмент с `seqovl`; **не** убирает оригинал → добавляйте `--lua-desync=drop` после.
- `wssize` / `wsize` — уменьшить TCP-окно.
- `drop` / `pass` — отбросить/пропустить.
- **Оркестраторы** (`circular`, `repeater`, `stopif`, `condition`) живут в `zapret-auto.lua` (движок
  грузит её автоматически). `circular` меняет стратегию при провале: `--lua-desync=circular:fails=N`
  и далее каждый инстанс с тегом `strategy=K` (K с 1, без пропусков); требует `--in-range=` (кэш
  входящих RST). `final` на инстансе останавливает ротацию.

**Fooling — критичные правила (иначе «зелено, но не работает»):**

- **`tcp_ts` должен быть ОТРИЦАТЕЛЬНЫМ** (`tcp_ts=-1000`). PAWS отбрасывает пакет с более старым
  (меньшим) TSval, поэтому фейк должен быть «старее». Положительный `tcp_ts` — баг: фейк выживает и
  роняет по PAWS уже **реальный** пакет.
- **`tcp_ts`/`tcp_ts_up` — пустышки без включённого TCP-timestamps.** Windows по умолчанию держит их
  в `allowed` (не `enabled`), поэтому Zapret2UI сам включает их на время сеанса (см. [§12](#12-tcp-timestamps)).
- **`badsum` ненадёжен за домашним NAT** — роутер дропает пакет с битой суммой ещё до DPI. На
  десктопе за NAT используйте `tcp_md5` (добавляет MD5-опцию, NAT-безопасно) или `tcp_seq` (badseq).
- **`multisplit`/`multidisorder` с `pos=1` в одиночку — полный no-op** (движок вырезает `pos=1`).
  Нужен реальный второй маркер: `pos=1,midsld`.
- **`seqovl` разный по смыслу:** у `multisplit` это **число байт** (сколько байт паттерна дописать);
  у `multidisorder` — **позиция-маркер**, которая должна быть меньше первой точки разреза.

**Позиционные маркеры:** `method, host, endhost, sld, midsld, endsld, sniext, extlen` + смещения
(`sld+1`, `midsld-2`).

**Блобы (важно!).** В комплекте едут `tls_clienthello_www_google_com.bin` и др. В стратегиях
используйте имена, заданные в глобалях: **`tls_google`**, **`tls_vk`**, **`quic_google`**,
**`quic_vk`**. Кастомных блобов вроде `tls5`/`tls7`/`tls1` из чужих сборок здесь **нет** — маппьте
любой `tlsN` на `tls_google`. Задаётся так: `--blob=tls_google:@{FILES}\fake\tls_clienthello_www_google_com.bin`.

**Мини-пример** (один профиль Discord-TLS с fake+split, схематично):

```
{WF_TCP} {WF_UDP}
--blob=tls_google:@{FILES}\fake\tls_clienthello_www_google_com.bin
--filter-tcp=443-65535 --filter-l7=tls {HOSTLIST:discord}
  --lua-desync=fake:blob=tls_google:tcp_md5:tcp_seq=-10000:repeats=6
  --lua-desync=multisplit:pos=1,midsld:seqovl=681:seqovl_pattern=tls_google
--new
--filter-udp=443-65535 --filter-l7=quic {HOSTLIST:discord} --payload=quic_initial
  --lua-desync=fake:blob=quic_google:repeats=11
```

Проверить строку целиком, не запуская движок, можно в предпросмотре команды (в интерфейсе). Помните:
`winws2` под админом, поэтому реальный тест — только запуском.

## 11. Встроенный Telegram-прокси

Telegram блокируют иначе, чем сайты, — часто **по IP**, и обычный desync не помогает. Поэтому в
программе есть **отдельный встроенный прокси**.

- **Что делает.** Поднимает локальный MTProto-прокси на `127.0.0.1:1443`; каждое соединение Telegram
  уводит к его дата-центрам по **WebSocket-TLS**, а при блокировке прямого пути — через **домены за
  Cloudflare**. Так связь переживает блокировку по IP.
- **Права.** Администратор **не нужен** (локальный слушатель + исходящий TLS). Работает независимо от
  основной кнопки обхода и даже когда окно свёрнуто в трей.
- **Как включить.** Тумблер Telegram (на «Главной» или вкладке Telegram) → «Открыть в Telegram» (
  прокси пропишется сам). Вручную: Telegram → Настройки → Данные и память → Прокси → Добавить →
  MTProto, вписать адрес/порт/секрет (кнопка «Копировать» рядом).
- **Порт и секрет.** Порт по умолчанию `1443` (если занят — берётся ближайший свободный, смена порта
  перезапускает прокси). Секрет (32 hex) сохраняется, чтобы ссылка `tg://proxy` не менялась между
  запусками. Ссылка: `tg://proxy?server=127.0.0.1&port=<порт>&secret=dd<секрет>`.

**Чем отличается от Flowseal/tg-ws-proxy.** Механизм тот же, но:

- Оригинал — отдельная программа на **Python**, собранная в самостоятельный `.exe` (PyInstaller).
- Здесь — **нативный порт протокола на C#**, встроенный прямо в приложение: без второго процесса,
  Python-рантайма и отдельного бинарника.
- Реализован транспорт `dd` (обфусцированный MTProto) — клиент локальный (loopback), поэтому FakeTLS
  (`ee`) не нужен и опущен. Рабочий путь сохранён: резолв через DoH и домены за Cloudflare (фронтинг)
  с пулом фронтов и временным отсевом «плохих». Код — MIT, с благодарностью Flowseal.

## 12. TCP timestamps

Часть стратегий (ts-fooling: `tcp_ts`/`tcp_ts_up`, на них держатся ALT10/ALT11) работает только если
исходящий TCP-пакет несёт опцию timestamp. Windows по умолчанию держит её в состоянии `allowed`,
которое **не гарантирует** timestamp на клиентских соединениях, — и тогда ts-fooling молча ничего не
делает, фейк не умирает и портит реальное соединение.

Поэтому Zapret2UI при **старте обхода** сам выполняет `netsh interface tcp set global
timestamps=enabled`, а при **остановке** возвращает прежнее значение. Делается автоматически, под уже
имеющимися правами администратора; вручную ничего трогать не нужно. В журнале это видно строками
«TCP timestamps: включены на время сеанса…» и «…возвращены в исходное состояние».

## 13. Память по сетям

Найденную рабочую стратегию программа привязывает к **отпечатку текущей сети** (по адресу шлюза/
роутера) и хранит в `settings.json` (`NetworkStrategies`). Когда вы возвращаетесь в знакомую сеть,
она предлагает/включает именно этот вариант, а не общий по умолчанию. Всё **локально**: в интернет
ничего не отправляется, IP-адреса не хранятся. На «Главной» это видно строкой «Стратегия для этой
сети: …».

## 16. Файлы и папки на диске

Всё лежит под `%LOCALAPPDATA%\Zapret2UI\` (программа никогда не пишет в Program Files):

```
%LOCALAPPDATA%\Zapret2UI\
├─ engine\                     движок и его данные
│  ├─ winws2.exe               сам движок (нужен админ)
│  ├─ mdig.exe, ip2net.exe     резолв доменов → CIDR для ipset
│  ├─ cygwin1.dll, WinDivert*  рантайм и драйвер
│  ├─ installed_version.txt    установленная версия движка
│  ├─ lua\                     zapret-lib / zapret-antidpi / zapret-auto и др.
│  ├─ files\                   блобы (fake\*.bin: TLS/QUIC/STUN ClientHello)
│  └─ windivert.filter\        windivert_part.*.txt (STUN, QUIC-initial)
├─ lists\                      хостлисты и ipset
│  ├─ youtube.txt, discord.txt встроенные списки доменов
│  ├─ exclude.txt              исключения (банки/госуслуги)
│  └─ ipset-discord.txt        собранные CIDR-подсети (после «Собрать IP-список»)
├─ logs\                       engine-YYYYMMDD-HHMMSS.log (вывод движка)
├─ tmp\                        временные загрузки
├─ presets.json               ваши стратегии
└─ settings.json              настройки (см. §14)
```

`settings.json` пишется атомарно (через temp-файл), чтобы сбой при записи не повредил настройки; если
файл всё же испортился, он сохраняется как `.bak`, а настройки сбрасываются на значения по умолчанию.

## 17. Решение проблем и коды ошибок

**Порядок действий при «не работает»:**

1. Переберите стратегии вручную сверху вниз (см. [§5](#5-встроенные-стратегии)).
2. «Подобрать лучшую», затем «Сгенерировать» на «Диагностике».
3. Настройки → **«Добавить в исключения»** (антивирус — самая частая причина).
4. Настройки → отключить **QUIC / HTTP-3**.
5. Смотрите **Журнал** — там причина, если движок не стартовал.
6. Блокировка по IP → VPN (или встроенный прокси для Telegram).

**Частые ситуации:**

- **Антивирус удаляет `winws2.exe` / «движок не найден».** Добавьте исключения, затем обновите/
  перекачайте движок (Настройки → обновление движка) или перезапустите программу.
- **«Не удалось запустить».** Причина — в Журнале. Чаще всего: запуск не от администратора, антивирус
  или недокачанный движок.
- **Работало и перестало.** Провайдер обновил фильтры → включите авто-починку или переподберите.
- **Discord: текст есть, голос молчит.** Стратегия `Discord — голос (QUIC-фейк)` или ALT10/ALT11.
  Голос идёт по UDP — на самых упрямых сетях провайдер режет его отдельно; тогда для стабильного
  голоса может понадобиться отдельный прокси/VPN именно под голос.
- **YouTube буферизует.** Отключите QUIC.
- **Всё зелёное, но сайт не открывается.** Другой механизм блокировки (ECH/IP) → другая стратегия/QUIC/VPN.

**Коды выхода движка (в Журнале):**

- `код 87` — недопустимый параметр. Частая причина — неизвестный глагол или отсутствующая библиотека
  (например, `desync function 'circular' does not exist`, если не загружена `zapret-auto.lua`).
- `код -1` / отрицательные — движок остановлен/убит (обычно при штатной остановке или смене стратегии).
- Прочие ненулевые — смотрите текст строки над кодом: он и есть причина.

## 18. Сборка из исходников и архитектура

**Сборка** (нужен .NET 9 SDK):

```powershell
# запуск для разработки
dotnet run --project ZapretUI/ZapretUI.csproj

# релизный self-contained single-file exe
dotnet publish ZapretUI/ZapretUI.csproj -c Release -o publish

# гейт качества — обе конфигурации должны быть 0 предупреждений / 0 ошибок
dotnet build ZapretUI/ZapretUI.csproj -c Debug   -warnaserror -nologo
dotnet build ZapretUI/ZapretUI.csproj -c Release -warnaserror -nologo
```

- .NET 9, `net9.0-windows`, x64, **WPF** + WinForms (только трей), **без сторонних NuGet-зависимостей**.
- `-warnaserror` — гейт: неиспользуемые поля/using ломают сборку (мёртвый код надо удалять).

**Архитектура** (MVVM, без DI-контейнера):

- **App / MainWindow** — точка входа, кастомное безрамочное окно, трей, глобальный обработчик падений.
- **MainViewModel** — единая VM: коллекции (стратегии, хостлисты, лог, диагностика) и команды
  (старт/стоп, подбор, генерация, ipset, переключатели).
- **Services** — `EngineService` (процесс `winws2`, разбор токенов), `PresetService`,
  `ComboStrategyCatalog`, `AutoSelectService`, `DiagnosticsService`+`NetProbe`, `IpsetService`,
  `MonitorService` (сторож авто-починки), `HostlistService`, `SettingsService`, `AutostartService`,
  `UpdaterService`, `TcpTimestampsService`, `TelegramProxyService`+`TgProxyCore`, `AppPaths`.

Полностью строение репозитория и правила разработки — в `CLAUDE.md`.
